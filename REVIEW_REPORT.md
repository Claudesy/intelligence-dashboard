## Security Findings & Remediations

### Summary

### Findings

#### M1: reactStrictMode disabled in next.config.ts
- **Severity**: Medium
- **Location**: `next.config.ts`
- **Description**: `reactStrictMode: false` disables double-render detection in development, reducing visibility of side-effect bugs in React components.
- **Remediation**: Re-enable with `reactStrictMode: true`. Note: this may expose latent issues with Socket.IO double-connection in dev; use `useRef` guard pattern.
- **Status**: Documented — requires testing before enabling

#### M2: Wildcard CORS origin in Socket.IO (development mode)
- **Severity**: Medium
- **Location**: `server.ts`
- **Description**: Development mode Socket.IO allows all origins. While guarded by `NODE_ENV` check, any misconfigured deployment without `NODE_ENV=production` would expose the WebSocket endpoint.
- **Remediation**: Add explicit origin allowlist even in development. Validate `NODE_ENV` on startup.
- **Status**: Documented

#### L1: EMR credentials stored in environment variables
- **Severity**: Low
- **Location**: `.env.example` (`EMR_USERNAME`, `EMR_PASSWORD`)
- **Description**: EMR login credentials for Playwright RPA automation are stored as environment variables. This is correct 12-Factor App practice, but the credentials should be rotated periodically.
- **Remediation**: Document rotation schedule in `SECURITY.md`. Consider using a secrets manager (Railway secrets / Vault).
- **Status**: Documented in SECURITY.md

#### L2: Session TTL is 12 hours
- **Severity**: Low
- **Location**: `src/lib/crew-access.ts`
- **Description**: 12-hour sessions are long for a healthcare system. 2026 best practices recommend ≤15 min for high-sensitivity systems, with refresh token rotation.
- **Remediation**: Consider reducing TTL to 4-8 hours or implementing sliding expiry with periodic re-authentication prompts.
- **Status**: Documented — requires UX consideration for clinical workflows

#### L3: Prisma Client logging in production
- **Severity**: Low
- **Location**: `src/lib/prisma.ts`
- **Description**: Without explicit log configuration, Prisma may log query parameters in development. Ensure production logging is restricted.
- **Remediation**: Explicitly configure `log: ['error', 'warn']` in production Prisma client.
- **Status**: Documented

## Remaining Recommendations

### High Priority
1. **Expand test coverage to ≥80%**: Currently ~10 test files for 300+ source files. Add unit tests for all API routes, library modules, and ML classifiers.
2. **Add ESLint configuration**: Replace `tsc --noEmit` only linting with ESLint (Next.js recommended config) for runtime safety and style enforcement.

### Medium Priority
3. **OpenAPI 3.1 specification**: Document all 70+ API endpoints in an OpenAPI spec. Use `zod-to-openapi` for automatic generation from existing Zod schemas.
4. **OpenTelemetry instrumentation**: Add distributed tracing for clinical workflows (EMR transfer, CDSS, telemedicine). Critical for debugging in production.
5. **Prisma connection pooling**: Add PgBouncer or Prisma Accelerate to handle connection limits under load, especially during peak clinical hours.
6. **API versioning**: Prefix all API routes with `/v1/` to enable future non-breaking evolution.
7. **reactStrictMode**: Re-enable after fixing Socket.IO double-connect patterns.

### Low Priority
8. **Session TTL reduction**: Review clinical workflow requirements and consider reducing session lifetime from 12h to 4-8h with graceful re-auth prompts.
9. **Prompt versioning**: Tag Gemini prompts in `audrey-persona.ts` and CDSS modules with version identifiers to track regression.
10. **Secrets rotation schedule**: Document EMR credential rotation schedule in operational runbook.
